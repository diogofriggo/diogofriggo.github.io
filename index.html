2017

Batman potential

Exact solutions to quantum problems are far and between and only exist for highly idealized or simple systems like a square potential. But computational physics knows no bounds! I wanted to show to my classmates that we could take the ideas we learned and apply it to any system. They were skeptical so instead of simulating a square potential like this: I simulated a batman potential: That's a probability wave interacting with a potential type Batman as I have dubbed it.
At each step Schrödinger's equation is being solved for each value of the domain

Fun fact: note how some probability is sneaking through to the other side of the square potential.

Classical mechanics says this is impossible. It's the macroscopic equivalent of a throwing a ball at a wall and expecting that it'll eventually go through unscathed if you try enough times. Not impossible in the quantum world!

The animation in the bottom depicts the same process in momentum space.
For the math geeks out there: that is a function! It doesn't appear so but it is piecewise defined so it is injective (how else could I draw a batman shape with a function?)
This also means that it's highly discontinuous, go solve that on pen and paper!
Coded in Python and plotted in Mathematica

2017

Fast Fourier Transform: Functional Programming

The FFT by itself can be used on pictures to reveal some really interesting physics.
Take a look at the cover where on the right you have the Fourier-transformed version of the left picture, produced by just applying an inverse fourier transform to each pixel on the left
The left picture is a microscopy image of graphene while on the right you see it in momentum space revealing several symmetries

The solution of the Schrödinger's equation for the Batman potential in the card above requires computing Fourier transforms
I ended up doing it in Python but I wanted to try it on a functional language so I picked up some Haskell and the resulting code is quite neat (5 lines) since the problem lends itself well to a functional style:

{% highlight haskell %} --factor that multiplies odd terms in the final sum f xs n = exp $ -(0:+2*pi)*n / genericLength xs --default case if the function gets called with --only 2 two arguments ffti [x,y] n = x + y * (exp $ -(0:+pi)*n) --otherwise apply the function recursively --splitting the sum into even and odd terms ffti xs n = ffti (evens xs) n + f xs n * ffti (odds xs) n --compute coefficients fft xs [] = [] fft xs (y:ys) = (ffti xs y):(fft xs ys) --example fft [0, 1, 4, 9] [0,1,2,3] --output (:+ means the complex part of a number) [14.0 :+ 0.0, -4.0 :+ 8.0, -6.0 :+ 0.0, -4.0 :+ -8.0)] {% endhighlight %}

Checkout me playing with it:

Everything we take for granted on higher level language becomes orders of magnitude more complex when talking directly to the machine
For instance, a simple sum in Python

{% highlight python %} a,b = 5,5 print(a + b) # 10 {% endhighlight %}

Is done roughly like this in assembly

{% highlight nasm %} addiu $v0,$0,5 #schedule read 1 syscall #read 1 add $t0,$0,$v0 #load read 1 addiu $v0,$0,5 #schedule read 2 syscall #read 2 add $t1, $0, $v0 #load read 2 add $t2,$t0,$t1 #sum add $a0,$0,$t2 #store sum addiu $v0,$0,1 #schedule print syscall #print addiu $v0,$0,10 #schedule exit syscall #exit {% endhighlight %}

2020

Report automation


The problem

Many engineers regularly having to write a 300-page report manually copying and pasting information from several sources into a word template document.
Other than the time spent, this approach frequently led to mistakes which delayed review and consequently delivery - the whole process could take up to 4 days

The solution

Developed an easy to use web app in Dash: now with a few clicks an energy analyst indicates where the many inputs are located, presses a button and within minutes gets a report with roughly 75% of all data filled in.
The remaining 25% were either very difficult to automate and thus didn't justify the effort to develop a solution to it or could not be done without human judgement.
A report can now be written in less than a day with far fewer mistakes.

The details

The core technology comes from the libxml2 C library which was ported to Python as lxml.
With it I can easily manipulate word's underlying OOXML, that is, I can change it programmatically to inject external information.
Knowing where to put the information is done via markers placed throughout the report template.
An earlier step of the code converts these markers into Word's Structured Document Tags They are basically xml with an identifier attached to it.
The neat thing about these tags is that I can replace the body text and they don't go away, this allows us to later update the report when needed, say with a different turbine layout Information to fill the report comes from:

  • Results from several internal excel tools
  • WindFarmer
  • Other word reports
  • The web
  • .csv and other textual formats

Inputs change their format or underlying representation quite frequently so this input-gathering aspect of the solution is a burden to maintain. Another team is currently addressing this with a single-source approach by further standardizing the workflow

Mar-Sep 2021

Minuano


The Problem

Many analysts with some level of programming skills produce useful tools often sharing them with others.
This is great but there are major drawbacks:

  • Not all analysts (current and future) will have easy access to these tools because they are scattered throughout the company
  • As each tool evolves and changes, different people end up with different versions of it
  • Every new tool needs to reinvent the wheel for the common bits like user interface, version management, integration with other tools, continuous integration, distribution... the list is long
  • In the long run it's very hard to maintain several tools living in completely separate environments

The Solution: Minuano

A centralized, self-contained and easily accessible dashboard of independent apps, abstracting away everything that is common to every app.

  • Every analyst has the latest version company-wide
  • Tool makers can focus only on their code and rely on Minuano to handle everything else
  • Straightforward to integrate a new tool into Minuano
  • Hosted on Azure DevOps, accessible company wide
  • Very intuitive to analysts (non-programmers) who previously had to manually run scripts and make sense of stack traces
  • Accessible from a desktop shortcut on each user's machine
  • Much easier to maintain

Trivia

Minuano is the name given to a strong, constant, sharp and dry wind that comes all the way from the Pacific, through the Andes and eventually reaches the south of Brazil. It has been named after an indigenous tribe that used to inhabit these lands - the Minuanos. The name was chosen to honor the ancient relationship of local inhabitants to wind which for today's inhabitants has taken the form of capturing energy from it with wind turbines!

2021

CFD post-processor


The problem

  • Several DNV offices across the globe perform CFD simulations on the same supercomputer
  • Each simulation can occupy hundreds of gigabytes, sometimes terabytes of disk space
  • Most of this information is redundant
  • Old projects must be archived on external hard drives only to be retrieved later when needed
    • This was a major problem during the pandemic because it meant someone had to go the office in-person to retrieve information

The solution

The solution came from an insight from my great colleague Leonardo Barriatto.
He realized that we could keep only a few isoheight slices of each simulation. When an intermediate isoheight was needed we could get it from simple linear interpolation without any significant loss of precision.

I was in charge of implementing the solution. I leveraged the whole Minuano infrastructure and built the solution as a new app to it. Integration was a breeze and the whole archiving system was made obsolete

Later stages of the tool

The tool has since grown to bring more and more features under its umbrella:

  • Generating point wind resource files at turbine locations taking stability-weighting into account
  • Analyzing speed-up plots
  • Computing cross-prediction errors
  • A very rich reactive UI enabling outstanding UX
  • Thorough user input validation that warns and prevents invalid use cases before running any calculation

Leonardo's skill as a project manager was fundamental to this project as it enabled a smooth workflow that allowed me to focus on technical aspects of the work while he worried about everything else.
The tool had significant success, it is used frequently by about a dozen CFD analysts company-wide.

  • The legacy codebase was hard to change and/or extend with new features

The solution

With funding from the CFD department, the legacy tool was rewritten from scratch in less than a month

  • Powered by Rust the new tool now produces the same results running in a user laptop and takes seconds instead of days
    • One sample case took 74010 seconds (supercomputer) in the legacy tool and 196 seconds (laptop) in the new one, that's a ~400x speed up
  • New tool extended with additional features
  • 3000+ lines of Rust code integrated easily into Minuano via PyO3 leveraging all the benefits of Minuano, in particular a façade for thorough validation of input data which then gets sent to Rust
  • Rust efficient use of memory was taken to the extreme to enable ingestion of gigabytes of CFD simulation grids, processing and writing results back to disk altogether in the seconds to few minutes range depending on project-size
  • Made heavy used of rayon to parallelize as much work as possible.
